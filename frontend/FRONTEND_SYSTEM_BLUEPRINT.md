# FRONTEND SYSTEM BLUEPRINT

Этот файл фиксирует, как дальше строится весь фронт проекта.

Цель:
- убрать хаос по страницам
- прекратить локальные “дизайнерские” исключения
- собрать единый конструктор страниц
- иметь одинаковый UI на desktop и mobile
- перестать возвращаться к одним и тем же вопросам по стилю и структуре

---

## 1. Главный принцип

Фронт больше не развивается как набор отдельных экранов.

Фронт строится как система:
- один `app shell`
- один набор page templates
- один набор table/filter/form primitives
- одна система токенов
- одна схема mobile/desktop renderers

Любая новая страница должна собираться из готовых блоков.

Запрещено:
- делать новый стиль под одну страницу
- делать локальный компонент, если он повторяет уже существующий паттерн
- смешивать desktop-логику и mobile-логику в одном монолитном JSX
- держать на странице одновременно 2 разных визуальных языка

---

## 2. Не делаем `frontend 2.0` как вторую независимую папку

Не создаём второй полноценный фронт рядом с текущим.

Почему:
- появится 2 состояния правды
- станет в 2 раза сложнее деплой
- начнётся расхождение компонентов, API-слоя и роутинга
- это только удвоит бардак

Правильный путь:
- текущий `frontend` остаётся единственным приложением
- внутри него вводится новый системный слой
- старые страницы постепенно мигрируют на него
- legacy удаляется только после перевода конкретной страницы

## 3. Целевая структура папок

Новая логика строится вокруг 5 слоёв:

```text
frontend/src/
  app/
    auth/
    catalog/
    pricing/
    sales/
    settings/

  components/
    shell/
    layout/
    navigation/
    data-display/
    forms/
    feedback/

  system/
    tokens/
    page/
    table/
    filters/
    forms/
    mobile/
    desktop/

  lib/
    api/
    auth/
    routing/
    utils/

  styles/
    globals.css
    tokens.css
```

### Что это значит

`app/`
- только бизнес-страницы и feature-экраны

`components/`
- UI-компоненты, которые реально переиспользуются

`system/`
- ядро фронтовой дизайн-системы и page constructor

`lib/`
- API, auth, helpers, route guards

`styles/`
- глобальные токены и общая база

## 4. Обязательная схема любой новой страницы

Каждая страница должна собираться одинаково:

1. `PageHeader`
- title
- subtitle
- right actions

2. `PageTabs`
- если есть разделы
- всегда один паттерн

3. `PageToolbar`
- поиск
- фильтры
- сортировки
- переключатели режима

4. `PageContent`
- таблица
- карточки
- редактор
- split layout

5. `PageActionDock`
- sticky actions
- save/apply/recalculate

6. `MobileActionDock`
- отдельный mobile-слой
- не через хаотичные padding-bottom

---

## 5. Единые рендеры для desktop/mobile

Для сложных страниц всегда делаем 2 renderer-компонента:

- `PageNameDesktop.tsx`
- `PageNameMobile.tsx`

Общий контейнер страницы:
- только держит controller/state
- выбирает нужный renderer

Пример:

```text
PricingSettingsPage.tsx
PricingSettingsDesktop.tsx
PricingSettingsMobile.tsx
```

Запрещено:
- пытаться одной вёрсткой обслужить сложный desktop и mobile
- прятать половину DOM через CSS и считать это адаптацией

Разрешено:
- общая логика
- общие hooks
- общие API
- разные renderers

## 7. Таблицы

Таблицы больше не рисуются “как получится”.

Единый стандарт:
- одна база таблицы
- единая типографика
- единые sticky headers
- единые paddings
- единый hover
- единые editable cells
- единый empty state
- единая pagination row

Любая новая таблица должна использовать:
- общий `DataTable` контейнер
- единые стили ячеек
- отдельные column presets

Таблица не должна:
- визуально разваливаться при добавлении колонок
- иметь новые случайные стили под каждый экран
- смешивать 2 разных подхода к редактированию

---

## 8. Каталог как единый навигационный слой

Каталог товаров и категорий должен быть единым источником навигации.

Это значит:
- `Категорийные затраты` используют один и тот же `CatalogBrowser`
- `Логистические затраты` используют тот же `CatalogBrowser`
- `Каталог` остаётся главным master-экраном

Не делаем:
- второй каталог для логистики
- третье дерево для ценообразования

Делаем:
- один `CatalogBrowser`
- разные режимы использования

---

## 9. Правила по mobile

Mobile не является “сжатым desktop”.

Для mobile:
- отдельный renderer
- отдельная scroll area
- отдельный action dock
- fullscreen sheets для сложного выбора
- bottom-nav должен учитываться архитектурно, а не отступами

Любой экран с:
- таблицей
- каталогом
- редактором

Должен проектироваться как mobile flow отдельно.

---

## 10. Правила по desktop

Desktop-экраны:
- строятся на широком canvas
- используют больше горизонтального пространства
- не должны раздуваться в бесполезные огромные карточки
- controls должны собираться в логичные строки

Если есть место в одну линию:
- элементы ставятся в одну линию

Если блок store-level:
- он должен читаться как toolbar/settings strip, а не как набор случайных box-виджетов

## 12. Правила разработки

Перед любым UI-изменением:

1. Посмотреть 2–3 похожие страницы
2. Выбрать существующий паттерн
3. Проверить, нет ли уже нужного компонента
4. Если паттерн повторяется 2+ раза:
   - вынести в общий компонент

Запрещено:
- править 1 страницу в отрыве от системы
- плодить CSS-классы без попытки унификации
- вводить новый таб-стиль, новый chip-стиль, новый card-стиль

## 14. Правило готовности страницы

Страница считается готовой только если:

- desktop и mobile собраны отдельными renderer-ами
- используются только системные primitives
- нет локального визуального “изобретения”
- таблица/фильтры/toolbar совпадают по языку с другими страницами
- состояние загрузки/ошибки/empty/save оформлено единообразно

Если это не так:
- страница не считается завершённой

## 15. Что делаем дальше

Следующий рабочий режим:

1. Не патчим страницы хаотично
2. Сначала доводим системные primitives
3. Потом по одной мигрируем страницы на конструктор
4. После миграции конкретной страницы удаляем legacy-код
